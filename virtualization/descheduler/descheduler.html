Descheduler for OpenShift Virtualization

Automated VM rebalancing via live migration — no downtime, no manual intervention

OpenShift 4.17+ KubeVirt Live Migration LongLifecycle KubeVirtRelieveAndMigrate

Overview

The Kube Descheduler Operator periodically evicts pods that violate placement rules or sit on overutilized nodes. For VMs, eviction does not kill the workload — KubeVirt intercepts and converts it into a live migration.

ℹ️
Key mechanism Descheduler issues a standard pod eviction. KubeVirt's PodDisruptionBudget intercepts it and creates a VirtualMachineInstanceMigration CR. The VM moves to a healthy node with <1s interruption.

LongLifecycle

  • Signal: CPU/Memory requests
  • Interval: every 1800s (30 min)
  • Threshold: <20% under / >50% over
  • Simple, broadly compatible
  • Cannot detect noisy neighbours

KubeVirtRelieveAndMigrate

  • Signal: PSI (Pressure Stall Info)
  • Interval: every 60s
  • Detects actual CPU/memory/IO contention
  • Taints hot nodes → steers new VMs away
  • Background eviction prevents oscillation

Prerequisites

  • Install Kube Descheduler Operator → namespace openshift-kube-descheduler-operator
  • VM spec.template.spec.evictionStrategy: LiveMigrate
  • VM spec.runStrategy: Always or RerunOnFailure
  • VM must be migratable (shared RWX storage, no host devices, no hugepages)
  • For KubeVirtRelieveAndMigrate only: PSI metrics enabled in kernel via MachineConfig and exported via Prometheus

Thresholds & Eviction Profiles Explained

Two settings that most VM deployments get wrong: devLowNodeUtilizationThresholds controls when to evict. EvictPodsWithPVC + EvictPodsWithLocalStorage control whether virt-launcher pods are even eligible for eviction.

devLowNodeUtilizationThresholds

Sets the underutilized / target threshold pair for LowNodeUtilization strategy. Nodes below underutilized = migration target. Nodes above target = evict candidate. Nodes between = untouched.

ValueUnderutilized (target)Overutilized (evict from)Use When
Low < 10% > 30% Stable cluster, only correct extreme imbalance
Medium default < 20% > 50% Most clusters — balanced rebalancing
High < 40% > 70% Nodes consistently imbalanced but rarely hit 50%

Node utilisation — Medium threshold example

EvictPodsWithPVC + EvictPodsWithLocalStorage

🚫
DefaultEvictor blocks virt-launcher pods by default. Without these profiles, descheduler silently skips all VM pods. Error: pod has local storage and is protected · pod with PVC is protected

virt-launcher pods always carry both protected volume types:

virt-launcher componentVolume typeBlocked by defaultProfile to unlock
VM disk (DataVolume / PVC)PersistentVolumeClaim✅ YesEvictPodsWithPVC
Virtio socket directoryEmptyDir (local)✅ YesEvictPodsWithLocalStorage
Cloud-init dataEmptyDir (local)✅ YesEvictPodsWithLocalStorage

🔒
Eviction ≠ data loss. These profiles only remove DefaultEvictor's blanket block. The eviction still goes through KubeVirt PDB → live migration. VM disk (PVC) stays on shared storage and is accessible from target node. EmptyDir is recreated on target.
⚠️
Local-only storage (no shared backend). If VM disk is on node-local storage (not Ceph / NFS / iSCSI), eviction passes but migration fails — disk not accessible from target node. Use evictionStrategy: LiveMigrateIfPossible to handle gracefully.

Scoping Descheduler to VMs Only

By default, descheduler targets any eligible pod on overutilized nodes — including regular app pods. Three options to restrict eviction to VM (virt-launcher) pods only.

OptionMechanismBest When
1. Namespace filter Only target listed namespaces VMs live in dedicated namespaces
2. Priority class Skip pods at or above priority threshold VMs mixed with apps in same namespace
3. Annotation opt-out Annotate regular pods to be skipped Few apps to protect, many VM namespaces

Option 1 — Namespace Filter

Simplest approach. Descheduler ignores all pods outside listed namespaces. Works cleanly when VMs are isolated in dedicated namespaces.

💡
Best practice: create dedicated namespaces per VM project (e.g., prod-vms, dev-vms) and list them here. Regular app namespaces are automatically excluded.
apiVersion: operator.openshift.io/v1
kind: KubeDescheduler
metadata:
  name: cluster
  namespace: openshift-kube-descheduler-operator
spec:
  managementState: Managed
  deschedulingIntervalSeconds: 1800
  mode: Automatic
  profiles:
    - LongLifecycle
  profileCustomizations:
    devEnableEvictionsInBackground: true
    namespaces:
      included:          # only these namespaces — all others ignored
        - vm-workloads
        - kubevirt-vms
        - production-vms

Alternatively, exclude specific namespaces and leave everything else targeted:

  profileCustomizations:
    namespaces:
      excluded:          # protect these — target everything else
        - default
        - production-apps
        - monitoring

Option 2 — Priority Class Threshold

Descheduler skips pods at or above the threshold priority. Assign regular apps a high priority class — descheduler leaves them alone and evicts only lower-priority virt-launcher pods. 

# Step 1: Priority class for regular pods (protected — high value)
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: regular-app-protected
value: 1000000
globalDefault: false
description: "Regular app pods — protected from descheduler eviction"
# Step 2: Assign to regular Deployments
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      priorityClassName: regular-app-protected
      containers: [...]
# Step 3: KubeDescheduler — skip pods at or above threshold
apiVersion: operator.openshift.io/v1
kind: KubeDescheduler
metadata:
  name: cluster
  namespace: openshift-kube-descheduler-operator
spec:
  managementState: Managed
  mode: Automatic
  profiles:
    - LongLifecycle
  profileCustomizations:
    devEnableEvictionsInBackground: true
    thresholdPriorityClassName: regular-app-protected
    # descheduler evicts ONLY pods with priority < regular-app-protected
    # virt-launcher pods default priority = 0 → evictable
⚠️
virt-launcher pods inherit their VM's priorityClassName. Ensure VMs do NOT use regular-app-protected or they will be skipped too.

Option 3 — Annotation Opt-Out

Annotate regular pods/deployments you want protected. Descheduler respects the annotation and skips them. Useful when only a few apps need protection. 

# On individual Pod
apiVersion: v1
kind: Pod
metadata:
  annotations:
    descheduler.alpha.kubernetes.io/evict: "false"   # descheduler skips this pod
# On Deployment (propagates to all pods)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        descheduler.alpha.kubernetes.io/evict: "false"
💡
Conversely, "true" forces eviction eligibility even on pods normally protected (e.g., pods with local storage or PVCs). KubeVirt uses this to allow virt-launcher eviction.

Combined — Namespace + Priority (Most Robust)

Namespace filter handles 95% of cases. Priority threshold handles edge cases where non-VM pods land in VM namespaces (operators, sidecars, etc.). 

apiVersion: operator.openshift.io/v1
kind: KubeDescheduler
metadata:
  name: cluster
  namespace: openshift-kube-descheduler-operator
spec:
  managementState: Managed
  deschedulingIntervalSeconds: 1800
  mode: Automatic
  profiles:
    - LongLifecycle               # or KubeVirtRelieveAndMigrate
  profileCustomizations:
    devEnableEvictionsInBackground: true
    namespaces:
      included:
        - vm-workloads             # only VM namespaces
        - production-vms
    thresholdPriorityClassName: regular-app-protected  # skip non-VM pods in those namespaces
✅
Result: descheduler only sees pods in VM namespaces AND below the priority threshold. In practice this means only virt-launcher pods — which KubeVirt then converts to live migrations.

Enabling PSI Metrics Required for KubeVirtRelieveAndMigrate

Pressure Stall Information (PSI) is a Linux kernel feature that measures % of time tasks stall waiting for CPU, memory, or I/O. It exposes real contention — not just utilisation — via /proc/pressure/. The descheduler reads these via Prometheus node_pressure_* metrics.

ℹ️
Requirements — cgroups v2 (default since OCP 4.14) + kernel arg psi=1. Fully supported from OCP 4.21. Works as tech preview on 4.17+.

MachineConfig — add psi=1 kernel argument

Apply one per MachineConfigPool. Triggers node drain + reboot via MCO — plan maintenance window.

Worker nodes:

apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  labels:
    machineconfiguration.openshift.io/role: worker
  name: 05-worker-kernelargs-psi
spec:
  kernelArguments:
    - psi=1

Master nodes (if monitoring control plane):

apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  labels:
    machineconfiguration.openshift.io/role: master
  name: 05-master-kernelargs-psi
spec:
  kernelArguments:
    - psi=1
⚠️
Prometheus memory impact. At 500+ containers, each Prometheus pod RSS increases ~1.3 GB (+42%) due to per-cgroup PSI metric cardinality. Monitor Prometheus memory after enabling.

Verify PSI is active on node

Run after MCO finishes rolling out the MachineConfig.

# Open debug shell on node
oc debug node/<node-name>
chroot /host

# Check PSI files exist and return data
cat /proc/pressure/cpu
cat /proc/pressure/memory
cat /proc/pressure/io

Expected output (values may vary):

some avg10=0.00 avg60=0.00 avg300=0.00 total=12345
full avg10=0.00 avg60=0.00 avg300=0.00 total=6789
✅
PSI active — file exists with some/full lines
❌
PSI not active — file missing or Operation not supported

Check MCO rollout status:

oc get mcp worker -o jsonpath='{.status.conditions[*].type}'
# Wait for: Updated=True, Updating=False

node_exporter automatically collects /proc/pressure/ when PSI is enabled. No extra config needed.

# Memory stall rate per node
rate(node_pressure_memory_stalled_seconds_total[5m])

Verify cgroups v2 is active

PSI requires cgroups v2. Default since OCP 4.14. Verify before enabling PSI. 

oc debug node/<node-name>
chroot /host

# Check filesystem type of cgroup mount
stat -f /sys/fs/cgroup | grep Type
✅
Type: cgroup2fs — cgroups v2, PSI supported
❌
Type: tmpfs — cgroups v1, must migrate to v2 first

Also check via node labels:

oc get node -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.nodeInfo.kubeletVersion}{"\n"}{end}'

# Or check MachineConfigPool for cgroupv2 kernel arg
oc get mc -o yaml | grep -A2 unified_cgroup
⚠️
Real-time kernel nodes may not support PSI. Exclude RT nodes from the PSI MachineConfigPool or use a separate MachineConfigPool for RT workloads.

Caveats

🚫
Non-migratable VMs block eviction. VMs with local storage, host devices, hugepages, or CDI DataVolumes not on shared storage cannot be migrated. Descheduler eviction will be blocked and retried next tick. Use evictionStrategy: LiveMigrateIfPossible for mixed environments.
🔁
Oscillation risk with LongLifecycle. Request-based balancing can move VMs back and forth between ticks if thresholds are tight. KubeVirtRelieveAndMigrate uses PreferNoSchedule taints to prevent this.
📋
PDB concurrency limit. KubeVirt PDB limits simultaneous migrations. If multiple VMs are evicted at once, some will queue. Tune spec.liveMigrationConfig.parallelMigrationsPerCluster in the HyperConverged CR.
